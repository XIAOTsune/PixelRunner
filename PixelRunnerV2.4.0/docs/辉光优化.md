# PixelRunner 辉光功能重做方案

更新时间：2026-04-20

## 结论先行

当前辉光功能不适合继续小修小补。我们已经验证过几条路线：

- Photoshop 原生多图层方案：最终图层相对稳定，但预览慢，交互体验不接近像素蛋糕。
- WebView CPU bloom 预览：交互方向正确，但算法效果差，容易出现白雾、鬼影、白衣被洗亮。
- CPU PNG 回贴：反复出现错位、拉伸、智能对象变换框问题，不适合作为当前最终应用路径。

因此后续目标不是“继续调现在这版参数”，而是重做辉光功能。

新方向：

```text
插件内实时预览：重做一套可控的预览算法
最终应用：优先使用 Photoshop 原生图层管线，保证图层落位稳定
算法目标：从简单 bloom 改为“高光源检测 + 发光层扩散 + 高光保护 + 风格化合成”
```

换句话说，第一阶段先做到“效果可接受、预览可交互、应用不乱位”。等这个版本稳定后，再考虑是否让最终应用也迁移到自研 CPU/Worker 管线。

## 当前问题复盘

### 1. 效果不佳

当前 CPU bloom 主要问题：

- 只按亮度阈值提取，高光区域太粗糙。
- 白衣、皮肤、浅色物体会被误认为发光源。
- 模糊层像白色雾片，不像真实光线扩散。
- 参数变化不够符合用户直觉，阈值、半径、强度的体感不稳定。
- 缺少高光保护，亮部容易死白。
- 缺少暗部保护，辉光容易把画面洗灰。

这说明辉光不是简单的“选亮部然后 blur”。要做得像像素蛋糕，需要先识别“应该发光的亮点”，再扩散，而不是让所有白色区域都发光。

### 2. 回贴不稳定

CPU PNG 回贴暴露了几个问题：

- `placeEvent` 会生成智能对象变换框。
- 透明 PNG 会触发 alpha bounds 相关逻辑，容易按可见区域而不是完整画布对齐。
- 即使加四角 `1/255` alpha 锚点，仍可能受到 Photoshop 置入/变换行为影响。
- `stretch` 会造成比例异常，`contain` 又可能留下定位和尺寸误差。

结论：在没有更底层的像素写入方式之前，不要把最终应用押在 `placeEvent + PNG` 路径上。

### 3. 预览和应用目标混在一起

之前我们试图让 CPU 管线同时解决：

- 实时预览
- 最终高质量输出
- Photoshop 图层写回
- 跨平台兼容

这个目标太大，导致每个环节都不够稳。后续必须拆开：

```text
预览只负责交互判断
应用只负责稳定落图
算法先负责效果正确
性能最后再优化
```

## 重做目标

### 用户体验目标

- 打开辉光面板后，插件内显示当前图像预览。
- 拖动滑块时，插件内预览能在 100ms 到 300ms 内反馈。
- 点击应用后，在 Photoshop 中生成稳定对齐的新辉光图层。
- 默认参数必须比当前版本明显更自然。
- 用户不需要理解复杂参数，也能得到可用效果。

### 效果目标

- 辉光主要来自灯、反光点、强高光边缘。
- 白衣和皮肤只允许轻微泛光，不应该变成大白雾。
- 暗背景不能被整体洗灰。
- 高光边缘要柔和，但主体纹理不能糊掉。
- 大半径应表现为空气感，不是糊成一片。

### 工程目标

- 预览不反复写回 Photoshop。
- 应用不走当前不稳定的 CPU PNG 置入路径。
- 每次实现一个可验证阶段，不再同时改算法、UI、回贴三件事。
- 每一阶段都要有测试样张和验收标准。

## 新架构

### 总体流程

```text
打开面板
  -> 捕获当前文档/选区预览图
  -> 插件内生成预览
  -> 用户调参数
  -> 插件内实时更新预览
  -> 用户点击应用
  -> Photoshop 原生流程生成最终辉光层
```

第一阶段最终应用继续使用 Photoshop 原生图层方案，不再使用 CPU PNG 回贴。

### 模块划分

```text
Glow UI
  管理面板、滑块、预览状态、应用按钮

Glow Preview Engine
  只负责插件内预览，不写 Photoshop

Glow Parameter Model
  统一解释强度、半径、阈值、曝光、光散射、风格

Glow Apply Adapter
  把同一套参数转换为 Photoshop 原生图层流程参数

Glow Test Harness
  用固定样张离线验证算法效果和耗时
```

## 算法重做方向

### 1. 不再只靠亮度阈值

新的发光源提取应至少结合三类信息：

```text
亮度：像素是否足够亮
局部对比：是否是局部高光，而不是大片白衣
饱和/颜色：是否像光源、反光、金属或边缘高光
```

初步公式：

```text
luma = perceptualLuma(rgb)
localMean = blur(luma, smallRadius)
localContrast = max(0, luma - localMean)
whitePenalty = smoothstep(whiteStart, whiteEnd, luma) * lowContrastMask

sourceMask =
  smoothstep(thresholdLow, thresholdHigh, luma)
  * (0.45 + localContrast * contrastGain)
  * (1 - whitePenalty * whiteProtect)
```

目的：

- 灯光、反光点、边缘高光更容易进入发光源。
- 大片白衣、皮肤高光会被压制。
- 阈值滑块有真实体感。

### 2. 分离“发光源”和“扩散层”

发光源不直接显示给用户，而是作为扩散输入：

```text
sourceMask -> sourceColor -> glowSmall -> glowMedium -> glowLarge -> combinedGlow
```

建议三层起步：

```text
small: 细边缘亮晕，半径较小，权重低
medium: 主辉光，半径中等，权重最高
large: 空气感，半径大，权重受“光散射”控制
```

避免再出现“清晰白色人形”。

### 3. 合成方式

插件内预览使用合成后的图，不再用透明 PNG 叠图：

```text
screen = 1 - (1 - base) * (1 - glow)
softAdd = base + glow * (1 - base * protect)
result = mix(screen, softAdd, styleWeight)
result = toneMap(result)
```

Photoshop 最终应用阶段可以继续生成辉光层并设为 `screen`，但图层内容必须是纯 glow，不包含原图。

### 4. 参数解释

第一版参数保持现有 UI，但内部重新解释：

```text
强度：最终 glow 增益
半径：扩散尺度，不直接等于 blur 半径
阈值：发光源筛选严格程度
饱和度：glow 层色彩保留程度
亮度偏置：发光源曝光补偿
风格：一组参数预设，而不是单独滤镜
```

风格建议：

- 自然：低强度、保护白衣、只做高光边缘。
- 柔和：扩散更宽、对比更低、适合人像柔光。
- 梦幻：大半径散射更强、允许更明显空气感。

## 最终应用策略

### 第一阶段：继续使用 Photoshop 原生应用

点击应用时：

```text
调用 photoshop.runToolAction({ action: "glow", ...params })
```

原因：

- 当前 CPU PNG 回贴路径不稳定。
- Photoshop 原生流程虽然慢，但落位相对可靠。
- 先把效果参数和预览体验打磨出来，再替换最终应用路径。

### 第二阶段：探索 Photoshop imaging 写入

如果后续必须自研最终输出，需要优先研究：

```text
Photoshop imaging.putPixels
新建像素图层后写入指定 bounds
避免 placeEvent / 智能对象 / PNG 置入
```

只有确认能直接写像素并稳定对齐后，才考虑恢复 CPU 最终应用。

### 暂停路径

以下路线暂停：

- CPU 透明 PNG 回贴。
- 黑底 PNG + screen 回贴。
- `placeEvent` 智能对象置入作为辉光最终应用。
- 继续微调当前 CPU bloom 试图达到像素蛋糕效果。

## 预览策略

### 预览输入

打开面板时捕获当前文档/选区：

```text
maxDimension: 1280
quality: 90+
```

如果用户切换文档、选区或图层，需要重新捕获。

### 预览输出

插件内显示：

```text
base + previewGlow 合成图
```

不写回 Photoshop。

### 预览性能

第一版可以继续主线程 Canvas，但需要做到：

- 参数变化 debounce 120ms 到 220ms。
- 每次预览带 jobId，过期结果不回写。
- 先只实现 1280 长边。

第二版再考虑 Worker：

- 亮度图和 sourceMask 放 Worker。
- blur 和合成放 Worker。
- UI 线程只显示结果。

## 验收样张

重做前先固定测试集，不能只凭一张图判断。

必须包含：

- 暗背景人像 + 手持光源。
- 白衣人像 + 高光边缘。
- 金属/玻璃反光。
- 室内灯光或霓虹。
- 低对比柔光图。

每张图都要输出：

- 原图。
- 显示发光范围。
- 自然风格结果。
- 柔和风格结果。
- 梦幻风格结果。

## 验收标准

### 参数验收

- 阈值从低到高，发光范围必须肉眼明显变小。
- 半径从低到高，扩散范围必须肉眼明显变大。
- 强度从低到高，亮度变化必须明显但不能爆白。
- 风格切换必须有可描述差异。

### 效果验收

- 白衣不会变成整片发光源。
- 皮肤不会出现大片白雾。
- 手持灯、金属反光、边缘高光能产生自然亮晕。
- 暗部不整体抬灰。
- 结果不出现明显脏边、断层、方块感。

### 工程验收

- 拖动滑块不卡死 UI。
- 预览不写回 Photoshop。
- 点击应用后图层与原图对齐。
- 取消预览不会留下临时图层。
- 出错时有明确提示，不产生半成品图层。

## 分阶段计划

### 阶段 0：停止失败路径

- 保留当前分支记录。
- 文档明确 CPU PNG 回贴暂停。
- 应用路径回到 Photoshop 原生流程。
- 不再继续调当前 CPU bloom 作为最终效果。

### 阶段 1：建立算法测试台

目标：不依赖 Photoshop UI，在浏览器或 WebView 中快速测试辉光算法。

任务：

- 准备固定样张目录。
- 写一个简单的 Glow Lab 页面或调试模式。
- 支持显示 sourceMask。
- 支持导出预览结果。
- 支持记录每次耗时。

验收：

- 能在同一张图上快速对比参数。
- 能明确看到 sourceMask 是否选错区域。

### 阶段 2：重做发光源检测

目标：解决白衣、皮肤、浅色区域误发光问题。

任务：

- 实现 luma map。
- 实现 local mean / local contrast。
- 实现 white area protection。
- 实现 sourceMask 可视化。

验收：

- 手持灯、金属反光进入 mask。
- 白衣大面积区域不进入或只轻微进入 mask。

### 阶段 3：重做多尺度扩散

目标：让辉光像光扩散，而不是白色雾片。

任务：

- 实现 small/medium/large 三层 glow。
- 半径映射到多层尺度。
- 光散射控制 large 层。
- 风格预设控制权重。

验收：

- 半径变化明显。
- 大半径有空气感。
- 小半径能保留亮边。

### 阶段 4：重做合成与保护

目标：避免过曝、发灰、脏边。

任务：

- 实现 screen/softAdd 混合。
- 实现高光保护。
- 实现暗部保护。
- 实现色彩保护和着色。

验收：

- 白衣不死白。
- 暗部不洗灰。
- 色彩不明显发脏。

### 阶段 5：接入现有辉光面板

目标：让新预览算法替换现有 CPU 预览。

任务：

- 保持现有 UI 不大改。
- 参数接入新的 Glow Parameter Model。
- 应用按钮继续调用 Photoshop 原生流程。
- 增加“显示发光范围”调试开关。

验收：

- 插件内预览稳定。
- 应用结果与预览方向一致。
- 不再出现白色鬼影。

### 阶段 6：研究最终应用重做

目标：评估是否能完全摆脱 Photoshop 原生多图层方案。

任务：

- 调研并实测 Photoshop imaging 是否支持直接写入像素。
- 如果可行，做最小像素写入 demo。
- 如果不可行，继续保留 Photoshop 原生最终应用。

验收：

- 只有在定位 100% 稳定时，才允许恢复自研最终应用。

## 当前代码处理建议

短期建议：

- 保留 `src/webview/glow-cpu.js` 作为实验代码，但不要视为最终算法。
- 保留插件内预览入口，但后续替换内部实现。
- 暂停 `placeResultFromUrl` 用于辉光最终应用。
- `src/host/photoshop/service.js` 中为辉光添加的 PNG 回贴逻辑不要继续扩展，后续可清理。

中期建议：

- 新建更明确的模块，例如：

```text
src/webview/glow/
  preview-engine.js
  source-mask.js
  pyramid-blur.js
  compositor.js
  presets.js
```

这样比继续把所有算法堆在单个 `glow-cpu.js` 里更容易调试。

## 下一步

建议下一步不再打包测试，而是先做阶段 1 的 Glow Lab：

```text
固定样张
显示原图
显示 sourceMask
显示预览结果
显示参数和耗时
```

只要 sourceMask 还选错，后面的辉光一定不好看。我们应该先把“哪些地方应该发光”解决，再谈扩散和合成。
