## 是否使用传统图像处理还是 AI mask

判断：两者都有，但辉光本身更适合用传统图像处理作为主体，AI/语义 mask 作为保护增强。

原因：

- `LensBlur/Kernel` 说明传统滤镜/核处理存在。
- 大量 AI 效果目录说明语义 mask 能力存在。
- `Shine` 配置不可读，不能确认它是否直接依赖 AI。
- 从效果目标看，辉光扩散天然适合传统多尺度 bloom；真正难点是“哪里该发光，哪里不该发光”，这部分可以被 AI mask 增强。

PixelRunner 建议：

```text
第一版：传统图像处理
  luma + local contrast + whiteFlat protection + skinLike protection + darkProtect

第二版：可选 AI mask 增强
  person / skin / cloth / background / hair / face masks
```

## 发光源和高光 mask 可能如何生成

像素蛋糕源码不可读，下面是基于产品效果和目录结构的推断。

它可能不是简单：

```text
luma > threshold
```

而更接近：

```text
候选高光 = 亮度高 + 局部对比高 + 颜色/饱和度符合光源或反光
保护区域 = 皮肤 + 大片白衣 + 白墙 + 暗部 + 低对比浅色区域
最终发光源 = 候选高光 * (1 - 保护区域)
```

PixelRunner 可实现自己的版本：

```text
lumaScore       = smoothstep(thresholdLow, thresholdHigh, luma)
contrastScore   = smoothstep(c1, c2, luma - localMean)
specularScore   = smoothstep(s1, s2, maxChannel - localMean)
whiteFlatMask   = highLuma * lowContrast * lowSaturation
skinLikeMask    = simpleSkinColorRange * midHighLuma
darkProtect     = 1 - smoothstep(darkLow, darkHigh, luma)

sourceMask =
  lumaScore
  * mix(0.55, 1.25, contrastScore)
  * mix(0.85, 1.15, saturationScore)
  * (1 - whiteFlatMask * whiteProtect)
  * (1 - skinLikeMask * skinProtect)
```

## 辉光扩散是否使用多尺度 blur / bloom / pyramid

没有源码级证据能确认 Shine 的扩散实现。但从近实时要求和通用做法判断，最合理的路径是多尺度扩散：

```text
sourceColor
  -> downsample pyramid
  -> each level blur
  -> upsample
  -> weighted combine
```

PixelRunner 当前 `src/webview/glow-cpu.js` 已经有 pyramid 雏形，问题主要不是 pyramid 方向错，而是 source mask 太粗、保护不足、合成保护不足。

建议重做时保留多尺度思路，但改成明确的三层视觉模型：

```text
smallGlow:
  高光边缘，半径小，权重低

mediumGlow:
  主辉光，半径中等，权重最高

largeGlow:
  空气感，半径大，受“光散射”控制
```

## 参数可能如何映射

建议 PixelRunner 的参数不要直接等于底层滤镜参数，而是映射到算法模型：

```text
强度:
  控制最终 glow 增益，不改变 mask 面积

半径:
  控制 medium / large glow 的扩散尺度

阈值:
  控制 sourceMask 的严格程度

曝光/亮度偏置:
  影响发光源检测的曝光补偿

光散射:
  控制 largeGlow 权重、最大半径和空气感

着色/饱和:
  控制 glowColor 的色彩保留，而不是直接改原图

风格:
  一组参数预设，例如自然、柔和、梦幻
```

风格可以这样映射：

```text
自然:
  source 严格，whiteProtect 强，largeGlow 弱

柔和:
  source 稍宽，mediumGlow 更宽，整体对比低

梦幻:
  source 更宽，largeGlow 更强，光散射更明显
```

## 像素蛋糕为什么能近实时

推断原因：

- 预览分辨率受控，不在全尺寸图上实时跑完整效果。
- 使用原生 C++/GPU/多线程效果引擎，而不是 JS 主线程全图循环。
- mask 和中间图可能会缓存，拖动滑块时只重算受影响阶段。
- 大半径扩散大概率在降采样图上完成。
- 有预览耗时统计调试开关，说明内部有性能预算。

PixelRunner 对应策略：

```text
第一版:
  1280 长边预览
  主线程 Canvas 可接受
  debounce 120ms 到 220ms
  jobId 防止过期结果回写

第二版:
  Worker 化
  sourceMask 缓存
  localMean / pyramid level 缓存
  参数变动时只重算扩散和合成

第三版:
  OffscreenCanvas 或 WebGL/WebGPU 可行性评估
```

## 可迁移内容清单

### 可以迁移的思路

- `sourceMask` 必须可视化。
- `protectMask` 必须可视化。
- 每次预览必须记录耗时。
- 算法测试台先于 UI 接入。
- 参数要通过统一模型解释。
- 多尺度扩散优于单层大半径 blur。
- 大片白色和皮肤需要保护，不应被简单亮度阈值选中。
- AI mask 可以作为增强，但不应阻塞第一版。

### 不应迁移的内容

- 像素蛋糕加密配置文件。
- 像素蛋糕资源素材。
- 像素蛋糕私有参数表。
- 像素蛋糕商业代码。
- 对加密配置进行逆向解密后的内容。
