# PixelRunner 2.4.0 AI优化功能方案

## 1. 目标

本文档用于定义 PixelRunner 插件内 `AI优化` 功能的产品定位、交互设计、配置方案和技术整合方式。

本功能的目标不是替代当前工作流应用，而是作为当前应用的辅助工具：

- 读取当前应用中已经填写的图片和提示词
- 调用一个单独的 RunningHub AI 优化应用
- 从 RunningHub 返回结果中解析 `.txt` 文本附件里的优化提示词建议
- 由用户确认后，再写回当前应用的提示词输入框

## 2. 当前决策

当前已确定的设计方向如下：

- `AI优化` 按钮放在工作台中 `预设` 按钮旁边
- 用户点击 `AI优化` 后打开弹窗
- 弹窗自动读取当前应用中已填写的图片和提示词
- 第一版只接入 `单图 + 文字` 模式的优化应用
- 优化结果先显示在弹窗中，不直接自动覆盖当前提示词
- 用户手动选择将结果 `替换当前提示词` 或 `追加到当前提示词`
- RunningHub AI 应用返回结果会包含一张图片和一个 `.txt` 文本附件
- 插件侧忽略返回图片，只解析返回的 `.txt` 文本内容
- `AI优化应用 ID` 放在设置页的 `高级设置` 中
- 插件内置一个默认 `AI优化应用 ID`
- 用户可以在高级设置中修改为自己的 RunningHub 应用 ID
- 当前已确定的默认 `AI优化应用 ID` 为 `2042544874578251778`

## 3. 功能定位

### 3.1 功能角色

`AI优化` 是辅助工具，不是当前主应用本身。

用户心智应该是：

- 当前选中的应用负责真正出图
- `AI优化` 只负责帮助用户优化当前 prompt
- `AI优化` 不切换当前应用
- `AI优化` 不自动改写当前输入，除非用户确认

### 3.2 为什么第一版只做单图 + 文字

选择 `单图 + 文字` 作为第一版能力的原因：

- 和插件现有工作流最匹配
- 当前插件最容易自动读取的是一张图片和一段 prompt
- 比 `只有文字` 更有价值
- 比 `双图 + 文字` 和 `三图 + 文字` 更容易落地
- 后续如需扩展多图，可以在现有弹窗基础上增加参考图区域

## 4. 用户流程

### 4.1 主流程

1. 用户在工作台中选择一个正常出图应用
2. 用户已填写当前应用的图片和 prompt
3. 用户点击 `AI优化`
4. 插件弹出 `AI优化` 弹窗
5. 弹窗自动显示当前识别到的图片和 prompt
6. 用户补充优化需求
7. 用户点击 `开始优化`
8. 插件调用 `AI优化应用 ID` 对应的 RunningHub 应用
9. 弹窗显示优化结果
10. 用户选择：
   - `替换当前提示词`
   - `追加到当前提示词`
   - `关闭`

### 4.2 第一版不做的事

第一版不建议包含以下内容：

- 多图输入模式选择
- 自动负向词拆分
- 自动直接覆盖当前 prompt
- 多候选提示词结果
- 结果历史记录
- 与模板系统联动保存

## 5. 工作台设计

### 5.1 按钮位置

在当前工作台 prompt 字段对应区域，保持和 `预设` 按钮同级。

建议按钮区：

- `预设`
- `AI优化`

### 5.2 按钮状态

建议规则：

- 检测到可写入的 prompt 字段时，`AI优化` 按钮可用
- 未检测到 prompt 字段时，`AI优化` 按钮置灰
- 第一版严格要求图片存在
- 未检测到图片时，按钮置灰或点击后提示不可用

推荐第一版策略：

- 没有图片时直接不可用
- 提示文案：`当前应用未检测到可用图片输入，无法使用 AI优化`

## 6. 弹窗原型

### 6.1 弹窗定位

弹窗用于两件事：

- 确认输入内容
- 预览优化结果

不承担复杂应用编辑职责。

### 6.2 标题与说明

标题：

`AI优化提示词`

说明文案：

`基于当前应用中的图片和提示词生成优化建议，确认后再写入当前提示词输入框。`

### 6.3 弹窗结构

建议分为 3 个主要区域：

1. 当前内容
2. 优化设置
3. 优化结果

## 7. 弹窗区域详细设计

### 7.1 当前内容区

标题：

`当前内容`

包含以下模块：

#### 模块 A：当前图片

显示内容：

- 当前识别到的图片缩略图
- 图片来源字段名
- 图片尺寸信息

未识别到图片时：

- 显示空态提示
- 第一版下不允许继续优化

空态文案：

`未检测到当前应用中的可用图片输入。`

#### 模块 B：当前提示词

显示内容：

- 当前主 prompt 字段内容
- 使用多行文本框展示
- 默认自动填入
- 允许用户在弹窗内微调

建议标题：

`当前提示词`

#### 模块 C：附加需求

类型：

- 多行输入框

用途：

- 用户告诉优化应用自己想强化什么

示例 placeholder：

`例如：保留主体姿态，强化3D效果，提升层次感，减少背景杂乱`

建议标题：

`附加需求`

### 7.2 优化设置区

标题：

`优化设置`

第一版建议只保留少量设置项。

#### 设置项 A：优化强度

选项：

- `轻度整理`
- `平衡优化`
- `强化重写`

默认值：

- `平衡优化`

#### 设置项 B：输出语言

选项：

- `中文`
- `英文`
- `中英混合`

默认值：

- `中文`

### 7.3 优化结果区

标题：

`优化结果`

建议包含：

- 一个大的多行结果文本框
- 结果框支持手动编辑

#### 初始态文案

`点击“开始优化”后，这里会显示 AI 返回的优化提示词。`

#### 处理中文案

`正在根据当前图片和提示词生成优化建议...`

#### 成功态

显示：

- `优化后提示词`

第一版建议只展示一份最终结果文本，不强制拆分结构化字段。

原因：

- 当前选定的工作流输出形态更接近自然语言结果
- 第一版优先保障简单可用

## 8. 弹窗按钮设计

### 8.1 按钮排列建议

左侧：

- `关闭`

右侧：

- `开始优化`
- `追加到当前`
- `替换当前`

### 8.2 状态逻辑

初始状态：

- `开始优化` 可用
- `追加到当前` 禁用
- `替换当前` 禁用

优化成功后：

- `追加到当前` 可用
- `替换当前` 可用

### 8.3 按钮含义

#### 开始优化

- 调用 `AI优化应用 ID` 对应的 RunningHub 应用
- 使用当前图片、当前 prompt 和附加需求作为输入

#### 替换当前

- 用结果文本直接替换当前主 prompt 字段

#### 追加到当前

- 在当前主 prompt 末尾追加优化结果
- 若当前 prompt 不为空，建议使用换行分隔

#### 关闭

- 关闭弹窗
- 不修改当前输入

## 9. 回填策略

### 9.1 第一版回填范围

第一版建议仅回填主 prompt 字段。

不建议第一版直接处理：

- 负向词字段
- 参数字段
- 多 prompt 字段

### 9.2 主 prompt 字段识别优先级

建议识别顺序：

1. `prompt`
2. `positive_prompt`
3. 第一个 prompt-like 文本字段

### 9.3 图片字段识别策略

建议识别顺序：

1. 当前应用中第一个已填写的图片字段
2. 若没有已填写图片字段，则视为不可用

### 9.4 追加策略

推荐行为：

- 当前 prompt 非空时：
  - 原内容 + 换行 + 结果内容
- 当前 prompt 为空时：
  - 直接写入结果内容

## 10. 设置页高级设置设计

### 10.1 新增字段

位置：

`设置 > 高级设置`

新增字段：

`AI优化应用 ID`

### 10.2 输入框设计

标签：

`AI优化应用 ID`

placeholder：

`填写 RunningHub AI优化应用 ID`

说明文案：

`用于“AI优化”按钮调用。插件内置默认值，也可以替换成你自己的优化应用。`

### 10.3 可选按钮

建议增加一个小按钮：

`恢复默认`

作用：

- 将当前输入恢复为插件内置默认值

### 10.4 默认值策略

建议行为：

1. 插件启动时优先读取用户保存的 `AI优化应用 ID`
2. 若未保存，则使用内置默认值
3. 设置页始终显示当前生效值
4. 用户保存后覆盖默认值

当前默认值：

- `2042544874578251778`

## 11. RunningHub 优化应用整合方式

### 11.1 整合思路

`AI优化` 应用是插件内的专用工具应用，不出现在普通出图应用选择列表中。

它的角色：

- 专门服务于 `AI优化` 弹窗
- 仅在用户点击 `AI优化` 时调用
- 不替代当前主应用

### 11.2 为什么不作为普通应用暴露

原因：

- 用户容易误以为它是一个出图应用
- 会污染当前应用管理列表
- 功能定位会混乱

推荐方案：

- 插件内部单独维护一个 `AI优化应用 ID`
- 工作台通过内置逻辑调用它

## 11.3 RunningHub 返回结果约束

根据当前已确认的信息，RunningHub 平台侧的 AI 应用要求保留图片结果，因此该优化应用的返回形式不是“纯文本”，而是：

- 一张图片结果
- 一个 `.txt` 格式的文本结果

插件侧对这个功能的处理原则应明确为：

- 不使用返回图片
- 只解析返回的 `.txt` 文本内容
- 将解析后的文本作为 `AI优化` 的结果展示在弹窗中

因此，插件实现时不要把这个功能当成普通返图应用处理，而要把它当成“文本结果来自附件文件”的特殊工具应用处理。

## 12. 第一版推荐输入输出

### 12.1 插件输入给优化应用的内容

建议输入：

- 当前图片
- 当前 prompt
- 附加需求
- 优化强度
- 输出语言

### 12.2 插件期望的结果

第一版建议只拿一份主结果文本。

但结合当前 RunningHub AI 应用的约束，实际返回结构应视为：

- 图片结果：忽略
- `.txt` 文本附件：插件真正使用的结果来源

插件只要求最终获得：

- 可用于回填的最终提示词文本

即使后续工作流内部更复杂，插件侧第一版也只依赖 `.txt` 中的主文本结果。

### 12.3 文本解析要求

由于插件真正依赖的是 `.txt` 结果文件，因此实现时建议明确以下规则：

- 先从 RunningHub 返回内容中找到 `.txt` 文件地址或文本附件对象
- 下载并读取 `.txt` 内容
- 按纯文本处理
- 不要求第一版支持 JSON
- 不依赖返回图片内容

如果 `.txt` 内容中带有多余说明文字，第一版可以先整体展示全文，再由用户手动确认后写回当前 prompt。

## 13. 状态与异常文案

### 13.1 未配置 AI优化应用 ID

`当前未配置 AI优化应用 ID，请到设置页高级设置中填写。`

### 13.2 未识别到图片

`当前应用未检测到可用图片输入，无法使用 AI优化。`

### 13.3 未识别到 prompt 字段

`当前应用未检测到可写入的提示词字段。`

### 13.4 优化应用运行失败

`AI优化失败，请稍后重试或检查优化应用配置。`

### 13.5 返回结果为空

`优化应用未返回有效提示词，请检查工作流输出。`

### 13.6 未找到可解析的 txt 结果

`AI优化应用未返回可解析的 txt 文本结果，请检查工作流输出配置。`

## 14. 第一版范围控制

### 14.1 第一版必须完成

- 工作台增加 `AI优化` 按钮
- 设置页高级设置增加 `AI优化应用 ID`
- 支持内置默认 ID：`2042544874578251778`
- 弹窗自动读取当前图片和 prompt
- 调用单图 + 文字优化应用
- 从返回结果中解析 `.txt` 文本
- 结果显示在弹窗中
- 用户手动执行替换或追加

### 14.2 第一版不建议完成

- 多图支持
- 负向词自动回填
- 多候选结果
- 模板联动保存
- 结果历史记录
- 自动覆盖当前 prompt

## 15. 后续扩展方向

第二阶段可考虑：

- 支持负向词输出和独立回填
- 支持双图或三图参考图模式
- 支持多候选结果
- 支持结果一键保存为模板
- 支持模型适配选项

## 16. 验收标准

### 16.1 产品层验收

- 用户可以在工作台中看到 `AI优化` 按钮
- 用户可以在设置页高级设置中看到 `AI优化应用 ID`
- 插件存在默认 `AI优化应用 ID`：`2042544874578251778`
- 用户可替换该 ID

### 16.2 功能层验收

- 弹窗能够正确读取当前图片和 prompt
- 点击 `开始优化` 后能成功调用优化应用
- 插件能够从返回结果中正确找到并解析 `.txt` 文本
- 优化结果会先显示在弹窗中
- 结果不会自动覆盖当前 prompt
- 用户点击 `替换当前` 或 `追加到当前` 后，当前 prompt 被正确更新

### 16.3 体验层验收

- 用户可以清楚感知该功能是辅助优化，而不是主流程应用切换
- 用户不会因为误触导致当前 prompt 被自动覆盖
- 弹窗文案和状态反馈清晰
