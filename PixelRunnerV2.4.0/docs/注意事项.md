# PixelRunner 2.4.0 注意事项

## 1. 架构原则
- 当前版本已经切换到 `UXP Host Shell + WebView UI` 架构。
- 后续开发必须继续坚持前后端分离和模块化。
- 不要把复杂 UI 或业务逻辑重新塞回 Host。

### WebView 负责
- UI 展示
- 页面状态
- 表单交互
- 设置页交互
- 工作台交互
- 任务参数组织

### Host 负责
- 挂载 WebView
- bridge 通信
- 本地存储
- RunningHub 网络请求
- Photoshop / UXP 能力调用

## 2. 正式源码位置
- 正式 Host 源码目录：`src/host`
- 正式 WebView 源码目录：`src/webview`
- 正式构建入口：
  - `src/host-entry.js`
  - `src/webview-entry.js`

## 3. 必须遵守
- 不要直接修改 `dist/*.bundle.js`
- 不要把复杂逻辑重新塞回宿主层
- 不要回退到根目录旧源码做主开发
- 浏览器预览时直接打开 `app.html`
- 不要在浏览器里直接打开 `index.html`
- 功能对齐以 `PixelRunnerV2.3.3` 为准
- 不要再参考 `HHPS v1.3.1`

## 4. 根目录旧源码说明
- 根目录旧的 `app-*.js`、`index.js`、`host-photoshop.js` 不再是正式入口。
- 旧文件已归档到：`legacy/root-src`
- 后续改动必须优先落在 `src/webview` 和 `src/host`

## 5. 当前工作台设计
- 已取消手动“放回 Photoshop”按钮。
- 当前设计为任务完成后自动贴回 Photoshop。
- 自动贴回逻辑：
  - 记录任务发起前的源文档信息
  - 默认允许跨文档回到源文档
  - 如果源文档存在原选区，则按原选区 bounds 自动贴回并加遮罩
  - 如果源文档没有选区，则自动回贴到源文档
- 这条链路已经迁移并构建通过，但真实 Photoshop 行为仍需继续在插件环境中验证。

## 6. 当前设置页状态

- 已完成或已接入的内容：

  - 账户状态读取
  - 应用管理编辑区
  - 应用列表搜索 / 排序第一版
  - 应用远程解析链路第一版
  - 最近一次解析调试记录读取入口
  - 模板轻量编辑区
  - 高级设置折叠区
  - 环境诊断折叠区
- 最近已补齐：
  - 设置页主链路乱码文案清理
  - 应用解析成功后的诊断摘要反馈
  - 应用输入结构摘要增强
  - 模板编辑和模板选择器文案清理

## 7. 应用解析现状
- `2.4.0` 现在不再只是前端提取 `appId`。
- 已补齐 Host 侧远程解析链路：
  - WebView 调 `runninghub.parseApp`
  - Host 访问 RunningHub 解析接口
  - 自动归一化应用名、描述、输入结构
  - 解析结果回填到设置页编辑区
  - 保存后落库并同步到工作台
- 已新增解析调试存储：
  - `rh_last_parse_debug`
- 设置页环境诊断区可读取最近一次解析调试记录。
- 最近已增强：
  - 应用输入结构摘要会显示必填/选填、图像、提示词、选项、布尔项和字段预览
  - 解析成功后的状态反馈更明确
  - 浏览器预览模式下的限制提示更清晰
- 仍需继续完善：
  - 解析失败时更友好的界面反馈
  - 更贴近 `2.3.3` 的解析结果展示
  - 更稳的输入标签识别与字段提示

- 当前这类结构损坏问题已修复，但仍要警惕残留乱码和旧文案。
- 最近已修复顶部三个栏下方内容挤压标题栏的问题。

### 遇到乱码时必须这样处理
- 优先检查：
  - `app.html`
  - `src/webview/*.js`

- 不要只看 `release` 或 `dist`
- 不要直接手改 `dist/app.bundle.js`
- 必须修源码后重新执行：
  - `npm.cmd run build`
  - `npm.cmd run package:test`

5. 优先继续完善应用解析体验，不要退回只做前端壳子
6. 继续完善提示词模板体验和残留文案 / 乱码清理
7. 然后再继续提升图像输入链路与工程化

## 最新同步（2026-04-07）

### 当前接手结论
- 当前正式开发目录仍然只有 `src/webview` 和 `src/host`。
- 当前正式构建入口仍然只有 `src/webview-entry.js` 和 `src/host-entry.js`。
- 功能与交互对齐基线仍然以 `PixelRunnerV2.3.3` 为准，不再参考 `HHPS v1.3.1`。
- 浏览器预览时直接打开 `app.html`，不要打开 `index.html`。
- 不要直接修改 `dist/*.bundle.js`，必须修改源码后重新构建。

### 这轮之后必须知道的工作台状态
- 工作台界面已继续向 `2.3.3` 收敛，重点是节约纵向空间和减少无关信息。
- “选择应用”区域已改成更紧凑的结构：上部放“切换 / 刷新”，当前应用显示尽量只保留应用名。
- 输入参数区正在持续做减法，避免把用户不需要理解的工程信息暴露在界面上。

### 图像输入链路最新约束
- 图像输入应继续参考 `2.3.3` 的交互，不要再保留“先捕获、再绑定”的两段式工程化界面。
- 当前方向是：图像字段自身就是主要交互入口，用户直接在字段区域点击，从 Photoshop 当前选区捕获图像。
- Host 侧已经补上优先读取当前选区的逻辑；若没有选区，再回退到整个文档。
- WebView 侧应继续保持界面简洁，不显示“字段绑定”“上传方式”“dataUrl”这类用户无感知价值的信息。

### 执行任务区最新约束
- 运行区不应只围绕单个 `runningTask` 做展示。
- 当前已经开始支持“多个进行中任务”的状态维护与逐个取消，后续继续完善时不要退回旧的单任务心智。
- 任务区的后续设计目标仍是：视觉上更接近 `2.3.3`，但能力上保留“可取消任意进行中任务”。

### 当前仍然要继续推进的方向
- 继续把 `workspace.js` 的布局、密度、提示文案向 `2.3.3` 收敛。
- 继续清理 `templates.js`、`settings.js`、`workspace.js` 里的残留乱码和旧文案。
- 继续完善应用解析后的反馈体验，不要停留在当前第一版成功/失败提示。
- 继续保持前后端分离：复杂 UI、状态、表单、交互都放在 WebView，Host 只负责 bridge、存储、RunningHub、Photoshop 能力。

### 每次改源码后的动作
- 执行 `npm.cmd run build`
