Quick Start

快速上手教程

先把基础链路跑通，再慢慢扩展自己的工作流。下面这 3 步是最推荐的新手路径。

3 步上手
01

在 RunningHub 中拿到 API Key

先去 RunningHub 账户里拿到可用的 API Key。这里更推荐使用企业共享 API，后续需要充值时，直接充值钱包即可。

建议把 API Key 的来源先确认清楚，再开始配置插件，避免后面因为余额或权限问题反复排查。
02

在设置页保存 API Key、AI 应用与预设提示词

进入插件设置页面，先保存 API Key，然后添加适合你当前业务的 AI 应用，同时把常用的预设提示词也一并保存好。

这一步做得越完整，后面 Workspace 的使用成本越低，尤其适合反复跑相似任务。
03

在工作区选择应用、输入参数并运行

回到工作区页面，选择目标应用，输入或上传需要的参数，然后点击运行即可开始任务。

运行时重点看状态摘要和日志区，它们通常能最快告诉你问题发生在哪一步。

Troubleshooting

为什么运行失败？

大多数失败其实都集中在少数几个原因。按下面顺序排查，通常会比盲目重试更快。

优先排查

最常见的 4 个原因

1. 全能图片 Pro / 2 应用使用人数较多，比较拥挤

这类热门应用在高峰时段更容易排队、超时或者表现不稳定，所以偶发失败并不完全是插件问题。

2. 钱包没有钱，请先查询余额

如果钱包余额不足，任务可能直接失败。建议先确认 RunningHub 钱包是否还有可用余额。

3. 图片涉及违规内容，未通过云端审核

这类失败通常不是参数格式问题，而是内容在云端审核阶段被拦截了。

4. 关闭梯子或加速器

本插件使用国内网站服务器运行，梯子、加速器或某些网络代理环境有概率导致请求被屏蔽或行为异常。

建议排查顺序

先看余额

这是最容易忽略、也最常见的原因之一。

再看应用拥挤程度

尤其是全能图片 Pro / 2，热门时段容易挤。

然后检查图片内容

如果内容可能踩审核规则，建议先换一张图再试。

最后排查网络环境

把梯子、加速器、代理先关闭，再重新运行一次。

一个经验： 如果你看到的是“偶发失败”而不是“稳定必现失败”，优先怀疑热门应用拥挤或网络代理环境，而不是先怀疑插件本身。

FAQ

为什么有时候返图是上一张图的？

这个现象偶尔会出现，核心问题通常不是“你没换图”，而是网站侧有时命中了之前一张图的缓存。

缓存相关

可能原因

1. 网站有时会命中前一张图的缓存

所以你已经换了图，但实际返回时仍然有概率拿到上一轮图像相关结果。

2. 这不是每次都发生

它更像是偶发性的缓存命中，因此体感上会显得“玄学”，但本质上是有明确处理办法的。

解决方法

方法 A：点击刷新应用之后再运行

这是当前最直接、最轻量的处理方式，适合大多数情况。

方法 B：切换到其他应用，再切回当前应用运行

如果你怀疑当前应用状态还残留旧输入，这种做法往往更稳。

建议习惯： 如果你刚替换了图片，又正好在全能图片 Pro / 2 这类热门模型上继续跑任务，最好先点一次“刷新应用”再提交。

DIY Workflow

如何自己添加应用？

插件不是只能跑内置示例。你完全可以去 RunningHub 官网挑选更适合自己业务的图生图 AI 应用，再添加到设置页里。

可扩展

推荐路径

去 RunningHub 官网，在 AI 应用界面里寻找合适的图生图 AI 应用。找到合适的应用后，把对应链接或应用 ID 放进插件设置页进行解析和保存即可。

1 进入 RunningHub 官网

先去官网的 AI 应用列表里筛选你需要的方向。

2 寻找合适的图生图应用

优先找参数结构清晰、输出稳定、适合你场景的应用。

3 复制链接或应用 ID 回到插件保存

解析成功后，它就会出现在你的应用列表里，后续可直接复用。

挑应用时的一个小建议

不要一上来就追最热门的模型。先从输入要求简单、日志更清晰、返图更稳定的图生图应用开始，后面再慢慢扩展到更复杂的工作流。
